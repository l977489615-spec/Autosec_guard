Product Positioning

项目不再讲“云端漏扫 + 车端漏扫联动”，而是明确为边缘端本地验证工作站

为什么必须本机化
  • 车端漏洞验证依赖 CAN、PCAN、蓝牙、Wi-Fi Monitor、SDR、USB 等现场硬件。
  • 许多 PoC 需要实验室私有网、隔离台架或车载以太网，远端云扫描无法直接触达。
  • 高风险 PoC 可能造成 DoS、复位、总线注入或目标异常，需要现场审批与回滚预案。
产品边界
  • React UI、Flask API、PoC Worker、沙箱 Runner 和 SQLite 均在本机运行。
  • Local Runtime 只检测本机能力，不再作为远端节点控制台。
  • Agent Scan 是本机服务内的智能编排能力，不承担云端调度定位。

核心一句话：把“车辆现场安全验证”做成可安装、可审计、可打包交付的本地工作站。

User Workflow

从本机能力识别到漏洞证据沉淀，形成闭环验证流程

注册登录本地账号、JWT、本地用户级 AI 配置
能力探测Local Runtime 刷新硬件与系统能力
选择目标IP、蓝牙 MAC、CAN 接口、Wi-Fi 接口、RF 频点
执行 PoCManual Scan、Global Scan 或 Agent Scan
报告审计历史记录、证据、攻击图、物理影响、报告

流程特点

流程不是单纯“扫端口”，而是先判断本机是否具备执行条件，再把 PoC 元数据、目标参数、风险审批、沙箱执行、证据归档串成一个可复查的车辆安全验证链条。

系统架构04
Local Runtime

本机能力探测决定哪些 PoC 具备执行条件

探测输入

  • 系统命令：lsusb、ip、iw、bluetoothctl、hciconfig、hackrf_info、rtl_test
  • python-can 配置、PCAN 字符设备、SocketCAN 接口
  • USB 挂载候选、Wi-Fi 接口、蓝牙控制器、SDR 状态

归一化输出

  • usb：USB / USB Gadget / 物理接口验证
  • can / pcan：CAN、UDS、OBD、重放、注入
  • wifi / bluetooth / sdr：无线、RF、TPMS、GPS、V2X 类验证

能力探测的价值是把“能不能执行”前置，减少误触发、误报告和不具备硬件条件时的无效测试。

PoC 覆盖面06
Safety Model

高风险 PoC 不是靠前端提示，而是前后端双层审批与沙箱执行

静态画像

poc_security.py 通过 AST 提取 PoC 类元数据，识别 severity、protocol、required_params、destructive_level 和 is_disruptive。

审批拦截

前端对高风险 PoC 做显式确认，后端再次根据 destructive_level / is_disruptive 执行强校验，避免绕过 UI 直接调用。

沙箱边界

sandbox_runner.py 约束 CPU、内存、输出大小、文件句柄和访问目标，并把日志、错误、证据和 trace_id 结构化保存。

安全控制07
Agent Scan

多 Agent 协作是本机智能编排能力，不是云端调度能力

Recon
调用 scan_ports、get_topology，输出服务列表、拓扑和攻击向量。
Planner
基于侦察结果生成无代码执行序列，避免直接进入破坏性动作。
Decision
调用 list_pocs、check_safety，过滤可用资源并补齐参数与安全策略。
Weaponize
针对未知服务生成协议感知型动态探测插件，并包装成合规 PoC 类。
Execute / Reflect / Assess
沙箱执行、失败反思恢复、生成符合 ISO 21434 / UN R155 语境的安全评估报告。
多 Agent 流程08
Evidence Model

证据、攻击图、物理影响和报告共享统一执行结果模型

统一 Finding 实体

  • trace_id：贯穿执行链路
  • poc_id / poc_name：关联具体验证脚本
  • target_ip / parameters：还原测试条件
  • severity / domain / evidence：支撑报告与图谱

下游消费

  • Scan History：历史审计与批量删除
  • Assessment：结构化安全报告
  • Attack Graph：攻击路径可视化
  • Physical Impact：物理影响评估

报告不是事后拼文本，而是从执行证据模型中派生出来。

Patent-Friendly Angles

更适合从现有代码中提炼的专利表达方向

A 篇：多 Agent 车辆漏洞验证编排

重点写 Recon → Planner → Decision → Weaponize → Execute → Reflector → Assessment 的阶段化协作，以及失败反思、动态探测插件包装、证据回流报告。

B 篇：能力感知的本机 PoC 安全执行

重点写本机能力探测、PoC 元数据安全画像、破坏性等级审批、沙箱执行、证据模型和本地报告生成。它不依赖 A 篇的 Agent 流程，也不涉及云边联动。

两篇的边界：A 写“智能编排决策链”，B 写“边缘端安全执行控制链”。
