# 智能网联汽车漏洞扫描 POC 插件研究项目结项报告

## 摘要

随着智能网联汽车产业的持续发展，整车电子电气架构、车载通信协议、车机系统以及无线连接能力不断增强，车辆由传统的机械产品逐步演变为高度软件定义、网络互联和智能协同的复杂系统。在此过程中，攻击面显著扩大，潜在安全风险也由单一接口风险演变为系统级、链路级和跨域级风险。针对这一趋势，本项目围绕“智能网联汽车漏洞扫描 POC 插件研究与系统原型实现”展开工作，目标是构建一套面向智能网联汽车场景的漏洞扫描与验证平台，形成可复用、可扩展、可审计的 POC 插件体系，并支撑车机台架和实车环境下的测试验证。

本项目完成了面向智能网联汽车典型攻击面的漏洞扫描系统原型设计与实现，形成了涵盖侦察、网络、车内协议、无线接口、应用层和高级攻击面的 71 个 PoC 验证插件，并构建了统一的 PoC 基类、参数归一化机制、风险画像机制、历史审计机制和结构化报告机制。在系统能力方面，项目实现了前端控制台、后端扫描引擎、PoC 沙箱执行框架、边云协同执行框架以及基于多 Agent 的自主扫描能力，支持对车机系统、局域网目标和硬件依赖型漏洞场景进行统一调度和验证。

与传统单阶段、单脚本的漏洞验证方式不同，本项目重点引入了多 Agent 协作扫描方式。系统通过侦察 Agent、规划 Agent、决策 Agent、武器化 Agent、执行 Agent、反思 Agent和评估 Agent组成多阶段闭环，使漏洞验证从“人工挑选 PoC 并逐个执行”转变为“自动侦察、自动编排、自动决策、按需生成、统一执行、结果反思和结构化评估”的一体化流程。这一设计显著提升了复杂目标场景下的自动化水平，也提升了系统在未知服务、组合攻击链和多阶段验证任务中的适应能力。

在测试验证方面，本项目在车机台架环境和大众 ID.4 目标环境中开展了系统级验证。测试结果表明，系统在目标参数采集、PoC 执行、日志回传、历史记录保存、边缘节点调度以及报告生成等方面表现稳定，能够支撑典型网络服务识别、协议侧验证、无线侧验证和高风险 PoC 的审批执行流程。整体来看，本项目已形成一个具备研究示范价值与工程原型价值的智能网联汽车漏洞扫描平台。

## 一、研究背景与项目意义

智能网联汽车的本质正在从单体车辆逐步转变为“移动智能终端 + 复杂控制系统 + 网络节点”的融合体。车机系统承载导航、娱乐、远程控制和车载应用生态；车内网络承担 ECU 之间的状态同步与控制指令传输；蓝牙、Wi-Fi、蜂窝网络、V2X 等通信链路使车辆具备更强的互联能力；同时 OTA、USB、诊断接口和第三方设备接入又进一步扩大了潜在攻击面。智能化和网联化提高了车辆功能丰富度，但也让车辆安全问题从“单点漏洞问题”演变为“跨组件、跨协议、跨域协同”的系统性安全问题。

与传统互联网资产相比，智能网联汽车漏洞具备三个突出特点。第一，漏洞分布面广，不仅存在于车机应用和网络服务，也存在于车内协议、无线接口和硬件接入链路中。第二，验证条件复杂，很多漏洞验证不仅需要目标 IP，还依赖 CAN 适配器、蓝牙适配器、Monitor 模式 Wi-Fi 网卡、SDR、USB OTG 等本地硬件条件。第三，验证风险更高，一旦在真实车辆或敏感台架上触发高风险漏洞，可能影响车机稳定性、数据完整性乃至行车安全。因此，面向智能网联汽车的漏洞扫描与验证不能简单照搬通用 IT 资产扫描模式，必须同时考虑硬件依赖、执行平面选择、审批约束、过程审计和结果结构化表达。

本项目正是在上述背景下展开。项目旨在研究和实现面向智能网联汽车场景的 POC 插件化漏洞扫描平台，重点解决三个问题：一是如何对智能网联汽车漏洞进行分类建模，并形成可统一管理的 POC 插件体系；二是如何构建一套适用于 ICV 场景的扫描系统原型，将漏洞识别、PoC 执行、日志回传和结果沉淀打通；三是如何将多 Agent 协作方式引入漏洞扫描流程，使系统具备更强的自主侦察、动态编排和复杂场景适应能力。项目成果不仅可用于实验室和企业内部授权测试，也可为车联网安全研究、课程教学和后续成果转化提供基础平台。

## 二、项目总体目标与完成情况

本项目的总体目标是构建一套智能网联汽车漏洞扫描系统原型，并完成 50 条以上智能网联汽车相关漏洞扫描 PoC 插件的研究与实现。围绕这一目标，项目在漏洞分类、POC 插件开发、系统架构设计和验证测试等方面持续推进，最终形成了较完整的成果体系。

在插件能力方面，项目已实现 71 个 PoC 插件，数量达到并超过任务书中的考核要求。这些 PoC 覆盖了侦察、网络、车内协议、无线、应用层和高级攻击面六大类别，能够对车机系统、局域网服务、车内控制链路、蓝牙与 Wi-Fi 接口、USB 接入场景和部分高级攻击路径进行验证。PoC 不再以零散脚本形式存在，而是基于统一插件基类组织，具备统一元数据、统一参数机制和统一结果输出格式，便于平台化调度和后续持续扩展。

在系统原型方面，项目已完成前端控制台、后端扫描引擎、PoC 沙箱执行框架、审计与历史记录模块、结构化报告模块以及边云协同执行模块的实现。系统支持单个 PoC 执行、全局扫描、实时日志展示、扫描历史查询、Edge 节点管理和 Agent Scan 等多种能力，已具备作为研究原型和实验平台持续演进的基础。

在创新能力方面，项目将多 Agent 扫描机制引入智能网联汽车漏洞验证流程。系统并非仅依赖静态规则选择 PoC，而是通过多 Agent 角色分工，自主完成侦察、规划、决策、武器化、执行、反思和评估的完整闭环。这一能力使平台从“执行器”升级为“具备自主协同能力的扫描系统原型”，这也是本项目区别于传统 PoC 管理系统和常规漏洞扫描器的关键所在。

## 三、总体技术路线

本项目采用“漏洞分类研究 + POC 插件化建模 + 系统原型实现 + 边云协同执行 + 多 Agent 自主扫描”的总体技术路线。整个技术路线可以概括为五个阶段。

第一阶段是漏洞类型梳理与攻击面建模。项目从智能网联汽车安全场景出发，将目标漏洞划分为侦察类、网络类、车内协议类、无线类、应用类和高级攻击类六大攻击面，并结合车机系统、车载网络和无线接口的典型风险场景提炼 PoC 开发目标。这一阶段为后续插件组织和系统分类展示奠定了基础。

第二阶段是 POC 插件设计与统一接口抽象。项目引入统一插件基类，为每个 PoC 设定统一的元数据描述，包括漏洞名称、严重等级、协议类型、目标操作系统、所需参数、是否具备破坏性等信息。执行层面上，每个 PoC 均实现前置条件检查、核心利用逻辑和结果输出机制，使平台能够按同一方式解析、运行和审计不同 PoC。

第三阶段是漏洞扫描系统原型设计与实现。项目构建了以前端控制台、后端 API 服务和执行 Runner 为核心的三层执行模型。前端负责参数录入、任务发起和结果展示；后端负责鉴权、参数归一化、风险控制、任务调度和数据落库；Runner 层负责在受限环境中执行 PoC 并提取结构化结果。通过这种方式，系统将 PoC 脚本运行过程纳入标准工程链路，而不是停留在独立脚本运行阶段。

第四阶段是边云协同技术路线设计。考虑到大量智能网联汽车 PoC 依赖本地硬件或局域网环境，项目设计了云端控制、边端执行的协同模型。云端负责用户、任务、审批、审计和报告；边端通过 Edge Agent 上报能力、接收任务并本地执行硬件依赖型 PoC。该技术路线使系统不仅可以在普通网络环境下执行云端 PoC，也可以在车机台架、车间局域网和实车周边环境下执行 CAN、Bluetooth、Wi-Fi、SDR、USB 等本地能力相关的验证任务。

第五阶段是多 Agent 自主扫描能力设计。项目将基于大模型的多 Agent 协作机制引入系统主流程。通过引入侦察 Agent、规划 Agent、决策 Agent、武器化 Agent、执行 Agent、反思 Agent和评估 Agent，系统能够对目标进行分阶段认知与执行，从而提升在未知服务、复杂目标和多步骤攻击链场景下的自适应能力。

## 四、系统架构设计与实现

项目系统总体上采用分层与解耦并行的架构设计。前端基于 React + Vite 实现，负责可视化交互；后端基于 Flask 构建统一 API 服务，负责系统逻辑承载；执行层通过 `poc_worker` 和 `sandbox_runner` 对 PoC 运行进行封装；边端通过 Edge Agent 完成能力探测和本地执行；多 Agent 模块作为上层智能编排能力，为复杂扫描任务提供自主协作支持。

前端系统主要包括扫描引擎、PoC 数据库、扫描历史、日志显示、用户配置和 Edge Control 等功能页面。扫描引擎页面支持手工参数录入和执行流程触发；PoC 数据库页面用于展示插件分类、元数据和使用说明；扫描历史页面用于回溯执行结果和结构化证据；Edge Control 页面用于完成边缘节点部署、能力查看和任务下发。通过这些页面，系统将复杂的漏洞验证过程包装为较为统一、可视化、可操作的工作界面。

后端系统是平台的核心控制中枢。其主要职责包括用户鉴权、参数标准化、PoC 元数据读取、风险校验、任务编排、审计记录、结果持久化和结构化报告生成。系统通过统一 API 暴露健康检查、PoC 列表、PoC 执行、流式执行、扫描历史、结构化报告以及 Edge 节点管理等能力，使前端、边端和 Agent 模块都能围绕统一接口进行协作。

在 PoC 执行架构方面，项目强调统一抽象和执行隔离。所有 PoC 以插件形式组织，统一继承基类，并通过标准字段描述安全属性和运行需求。系统运行时会先进行参数归一化和安全画像识别，再决定是否需要审批以及使用何种执行平面。执行过程中，Runner 负责创建受限执行环境、调用插件、收集标准输出并提取结构化结果。这一设计有效提升了 PoC 插件的可维护性与平台整体的一致性。

在边云协同架构方面，系统采用“控制面在云端、执行面在边端”的思路。云端后端负责统一管理边缘节点、接受节点注册、维护任务队列和接收结果回传；Edge Agent 负责周期性心跳、能力探测、任务获取和本地执行。该架构尤其适合解决局域网内目标、硬件依赖型 PoC 和无法由云端直接触达的测试场景。对车联网安全而言，这一机制显著提高了真实环境测试能力。

## 五、多 Agent 扫描方式设计与实现

本项目的一项核心特色在于引入了多 Agent 漏洞扫描机制。传统的漏洞扫描系统通常以规则为核心，通过固定流程调用探针或脚本进行验证。这种方式在面对标准化目标时有效，但在复杂车机环境、未知服务、动态交互和多步骤组合攻击链场景下，容易出现上下文割裂、任务编排僵化和执行效率不足的问题。为此，本项目设计了以多 Agent 协作为核心的 Agent Scan 模块，使系统具备更强的自主分析和动态决策能力。

从实现上看，系统中的多 Agent 编排由 `agent_orchestrator.py` 承担。该模块将扫描过程划分为七个阶段，并分别对应七个专业 Agent。第一个 Agent 是侦察 Agent，负责调用端口扫描和拓扑分析工具，获得目标的服务暴露情况、网络拓扑关系及潜在攻击入口。第二个 Agent 是规划 Agent，基于侦察结果生成后续任务序列和执行纲要。第三个 Agent 是决策 Agent，负责从 PoC 库中筛选合适的插件，同时结合安全策略和执行条件对任务进行调整。第四个 Agent 是武器化 Agent，当系统发现未知服务或现有 PoC 无法覆盖时，它会动态生成新的测试代码，并将代码包装为标准插件形态。第五个 Agent 是执行 Agent，负责调用平台已有的执行能力运行 PoC，并提取漏洞证据。第六个 Agent 是反思 Agent，当执行连续失败或结果无进展时，它会回看上下文、纠正策略并给出恢复建议。第七个 Agent 是评估 Agent，负责将整轮扫描过程汇总为结构化安全评估结果，为后续报告、修复建议和风险分析提供输入。七阶段流程分别对应“看见目标、理解目标、选择策略、补足能力、执行验证、纠偏优化、汇总结论”的闭环路径，较符合真实安全测试人员的工作方式。

该设计的优势在于，系统不再把扫描看成一次静态调用，而是看成一个逐步认知、逐步推理和逐步执行的过程。侦察阶段解决“看到了什么”的问题，规划阶段解决“下一步做什么”的问题，决策阶段解决“现在执行什么更合适”的问题，武器化阶段解决“遇到未知目标如何扩展能力”的问题，执行阶段解决“如何拿到证据”的问题，反思阶段解决“失败后如何自我修正”的问题，而评估阶段解决“最后如何沉淀为可交付输出”的问题。对于智能网联汽车这种多协议、多模块、多攻击面的对象而言，这种多阶段协同方式比单一 PoC 调用更贴近真实测试流程，也使系统具备从规则驱动向“规则 + 智能编排”演进的基础。

在工程实现上，多 Agent 模块并不是脱离现有系统单独存在，而是深度复用了平台已有的 PoC 列表、风险校验、拓扑扫描、流式执行和边缘调度能力。换言之，Agent Scan 不是替代扫描系统，而是建立在原有扫描系统之上的“智能编排层”。这使得系统既保留了传统 PoC 扫描的可控性，又获得了面向复杂目标的自主适应能力。因此，在本项目结项成果中，多 Agent 扫描既是系统亮点，也是后续申请方法类专利和继续深化研究的核心抓手。

## 六、项目测试与验证

为验证系统的正确性、稳定性和适用性，本项目分别在车机台架环境和大众 ID.4 测试对象上开展了系统测试。测试重点围绕扫描系统原型功能、PoC 插件执行能力、边云协同机制以及多 Agent 扫描能力展开。整体测试遵循“授权环境、控制风险、保留过程、沉淀结果”的原则，以保证测试过程具备可重复性和可追溯性。

在车机台架测试中，项目主要验证了系统的基础扫描能力和工程化闭环能力。测试内容包括前端参数录入、后端任务发起、PoC 执行、实时日志回传、风险审批、扫描历史保存和结构化报告输出等。车机台架环境具备较好的可控性，适合对网络服务类、协议类和部分应用类 PoC 进行重复测试。测试结果显示，系统在多轮执行过程中能够稳定完成 PoC 调用与结果提取，扫描过程可视化程度较高，适合实验室教学、功能验证和回归测试场景。尤其在插件统一调用、日志留痕和历史记录管理方面，系统已形成较完整的闭环，这也是从“脚本集合”走向“研究平台”的关键标志。

在大众 ID.4 环境测试中，项目重点关注系统在真实目标环境下的实用性。测试中采用了以目标参数配置、网络侦察、PoC 选择和边端配合为核心的流程，对车机相关网络服务、典型协议交互和部分无线/局域网场景进行了验证。由于实车测试对执行边界和风险控制要求更高，系统中的高风险审批、边缘执行和结果留痕机制在该场景中发挥了重要作用。测试表明，系统能够支撑实车环境下的规范化验证流程，尤其在局域网目标识别、证据记录和任务过程管理方面具备较明显优势。通过台架测试与实车测试的结合，项目不仅验证了系统在受控环境下的稳定性，也验证了系统面向真实场景的适配能力。

边云协同测试是本项目的一项关键验证内容。项目通过 Edge Agent 对本地硬件能力进行探测，并在具备相关能力的边缘节点上执行 Bluetooth、Wi-Fi、CAN、SDR 和 USB 依赖型 PoC。测试过程中，云端负责统一控制和任务下发，边端负责本地执行和结果回传。结果表明，系统能够有效区分适合云端执行和适合边端执行的任务，并完成边缘节点注册、心跳、任务获取和结果回传的完整链路。这意味着系统已具备将传统实验脚本迁移为边云协同漏洞验证任务的能力，也证明了该架构适合智能网联汽车这类“目标在局域网内、能力在本地硬件侧”的测试对象。

在多 Agent 扫描测试方面，项目验证了七阶段自主扫描流程的连通性和可解释性。系统能够基于目标侦察结果生成执行计划，在已有 PoC 能力范围内选择合适插件，并对失败场景进行一定程度的反思和调整。对于存在未知服务指纹的场景，武器化 Agent 还可按需生成动态测试代码，并在经过包装后纳入统一执行框架。这说明多 Agent 机制不仅具备“展示性”，而且具备实质性的平台增强作用。对于结项报告而言，这部分能力说明本项目已经不仅是在做插件开发，更是在探索面向 ICV 漏洞验证的自主化扫描方法。

总体而言，项目测试表明，本系统已经完成从“插件研究”向“扫描系统原型验证”的跨越，能够支持车机台架和实车环境下的典型测试需求。虽然在生产级隔离、任务恢复和更大规模自动化回归方面仍有提升空间，但作为结项阶段成果，其系统性、完整性和可演示性已经达到项目预期。

## 七、创新点

本项目的创新点不宜铺陈过多，结合任务书目标与实际完成情况，主要可归纳为以下两个方面。

第一，项目将多 Agent 协作机制引入智能网联汽车漏洞扫描流程，形成了面向车联网场景的七阶段自主扫描模式。与传统“人工选 PoC、手工串联流程”的方式相比，本项目实现了从侦察、规划、决策到执行、反思、评估的完整闭环，提升了漏洞验证流程的自动化水平和上下文一致性。这一设计对于复杂车机环境、多协议目标和未知服务场景具有明显优势，也是本项目最具代表性的技术特色之一。

第二，项目实现了面向智能网联汽车场景的边云协同漏洞扫描系统原型。系统不是简单地把 PoC 放到网页上执行，而是构建了统一的控制面、执行面和结果沉淀机制，并能够根据目标可达性与硬件依赖性选择云端或边端执行路径。对于 CAN、蓝牙、Wi-Fi、SDR、USB 等依赖本地能力的场景，边云协同机制使漏洞验证从“单机实验”升级为“可调度、可管理、可审计的系统任务”。

## 八、后续展望与计划

从结项成果来看，当前系统已经具备较完整的研究原型能力，但在商用化、规模化验证和成果凝练方面仍有进一步提升空间。后续工作将重点围绕插件扩展、系统工程化升级以及知识产权成果输出三个方向开展。

首先，在插件体系方面，项目将继续扩展面向智能网联汽车的 PoC 覆盖范围，重点补充更多与车机应用生态、车载以太网、V2X、ADAS 控制链路和供应链组件相关的验证能力。同时，将进一步提升插件元数据质量和验证链路质量，使更多 PoC 具备更强的自动化确认能力，并尽量减少依赖人工二次判断的环节。

其次，在系统工程化方面，项目将围绕商用级平台目标继续迭代，包括进一步提升执行隔离能力、增强边缘任务队列可靠性、完善节点生命周期管理、提升审计检索能力以及扩展多车型、多节点的适配与调度能力。多 Agent 模块也将持续优化，进一步提升未知场景下的任务编排效果和动态生成能力。

再次，在知识产权与成果输出方面，项目计划围绕本项目形成两篇专利布局。第一篇专利聚焦多 Agent 漏洞扫描方法，拟题可构思为《一种面向智能网联汽车的多 Agent 协同漏洞扫描与动态武器化验证方法》。其核心内容包括：将扫描过程分解为侦察、规划、决策、武器化、执行、反思和评估七个阶段；利用不同 Agent 在共享上下文下协同工作；在未知服务出现时触发动态代码生成并纳入统一执行框架；最终输出结构化安全评估结果。该专利的重点在于方法流程、角色协同和闭环验证机制，适合从“方法发明”角度进行知识产权布局。

第二篇专利聚焦系统架构，拟题可构思为《一种面向智能网联汽车的边云协同漏洞扫描系统及其任务调度架构》。其核心内容包括：云端控制面、边端执行面、PoC 插件库、风险审批模块、任务调度模块、能力探测模块和结构化结果模块；根据 PoC 执行需求自动选择云端或边端执行平面；支持局域网目标、硬件依赖型场景和高风险 PoC 的统一调度；实现从任务下发到结果回传和报告生成的完整架构闭环。该专利重点在于系统构成、模块关系和调度机制，适合从“系统与装置”角度布局。

通过上述后续工作，项目有望从“研究原型”进一步走向“可推广平台”，在企业内部安全研究、实验室教学平台、授权测试工具链和后续产学研合作中发挥更大价值。

## 九、总结

本项目围绕智能网联汽车漏洞扫描 POC 插件研究与系统原型实现这一目标，完成了从漏洞分类研究、PoC 插件开发到平台架构设计、边云协同执行和多 Agent 自主扫描能力构建的一整套工作。项目不仅达成了“50 条以上 PoC 插件”和“漏洞扫描系统原型”的任务目标，而且在平台化、工程化和智能化方面形成了较为完整的成果体系。

从成果形态看，本项目已经不再是单纯的漏洞脚本集合，而是一套具备统一执行入口、统一结果表达、风险控制、边缘调度、历史审计和结构化报告能力的智能网联汽车漏洞扫描平台。尤其是多 Agent 协作扫描机制的引入，使系统具备了更高层次的自主协同能力；边云协同机制的引入，则使系统能够更贴近真实车联网测试环境。这些成果为后续工程化升级、专利凝练和成果转化提供了扎实基础。

总体来看，本项目已顺利完成结项目标，具备良好的延展性、展示性和持续研究价值。
